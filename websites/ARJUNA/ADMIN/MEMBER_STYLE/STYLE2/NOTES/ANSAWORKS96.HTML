PART 3: INGREDIANTS

  • computing intelligence moves to edge
    • was: intelligent mainframe talking to dumb terms + disks
    • now: dumb network, clever terminals, clever apps (DBs) (disks)
  • same story in telephone systems
    • was: dumb phones, clever exchange
    • now: clever phones, dumb network, clever services

Currently we describe machine by the OS, in the future it will be by the browser we run - this will define the semantics. Browsers are duplicating OS functions - managing UI, cache management. This will promote the microkernel based operating systems - boot ROM of system. OS services e.g. persistence will be downloadable. CORBA orb acts like microkernel. Orbs may move down and sit above raw machine, providing common API on top of which services are downloaded. Feature interaction is a problem.

We also need service manager (network operating system), managing downloaded software components e.g. controlling services that are available to a downloadable component.

Web ideas: remembering routes traveled through the web and recognises when we start following it again. Automatic classifications, if I access an item, other items of the same category could be located for the user.

Evolution: Java app may be around for 100 years, encoding formats change. Reflective systems - apps can change internals e.g. smalltalk.

Intentional programming (Microsoft) - abstract syntax trees - high level data is still intact but can be mapped to different architecture. If all tools are IP equipped, different bits can be written in different languages.

Perhaps can move from AST back to new programming languages. This means keeping the meta-data with the application.

WebServer 2000

  • manage as a single system but spread out over many hosts
  • CORBA provides backbone
  • object wrapping - access to legacy apps, OLE SQL + CGI apps
  • session management - state, location in system, security, things already bought, accounting mechanisms, QoS
  • scripts - declarative descriptions of payment policies, management, access control, replication
  • page generator - generates pages based on script information

ReTINA - JEAN-BERNARD STEFANI, FRANCE TELECOM

Objectives

  • build a TINA compliant DPE (distributed programming environment)
  • complement CORBA for real-time & telecom specifics
  • validate DPE through demonstrators
  • influence standards: OMG telecom taskforce, TINA-C

Requirements

  • conformance: upward compatibility with CORBA2
  • modularity and Scaleability: an ORB everywhere
    • ORB on top of every piece of network hardware
    • including mobile phones - change interfaces - locale
    • modularity is essential
  • flexibility: protocols, bindings & policies
    • micro-ORB + special modules depending on environment
    • telecoms system have many protocols not just TCP/IP
    • indeed new protocols will appear on the web
    • plug in policies too
    • plug in new bindings - client-server not always suitable
    • e.g. add caching different caching strategies
    • these transparencies are essentially different bindings
    • many different binding models available
    • e.g. multi-cast or multi-point to multi-point comms
    • ORB currently gives 1 binding option & only 2 transports
  • real-time: resource control & kernel determinism
    • fine control of low level resources e.g. threads, memory etc.
    • KD: guarantees of well-known temporal behaviour

Technical Approach

  • small, portable DPE kernel
  • DPE services 'layered' on the DPE kernel
  • different DPE profiles in different environments
    • real-time profile - chorus
    • general purpose profile - standard UNIX
  • Scaleability: kernel ported on different micro-kernels configurations
    • 10K to 1Meg kernels
  • flexibility: remove bindings from the kernel
    • add binding libraries
    • adapter classes
  • real-time: generic resource control frameworks
  • bindings as object: access path to object
    • set of names
    • set of resources that are required to provide the path
    • possibly distributed objects
  • bindings created through binding factories
  • bindings with interfaces for specific control
  • CORBA extension: binding factories as object adapters
  • data management services in ReTINA DPE through O2 DBMS
    • fully fledged query language
  • query and persistence through O2 objects
  • O2 transactions embedded in CORBA-compliant transactions
  • generic connection management framework as a telecom specific DPE service
  • connections as specialised forms of bindings
