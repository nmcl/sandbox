Scaleable Referencing for the Web

5th International World-Wide Web Conference, Paris, May 1996

(Submission date: 29th January 1996)

Aims

  • To provide referential integrity guarantees for W3Objects (web objects)
  • To provide object migration transparency

Requirements

(random order brain dump noting Doug Seaton's comments)

Object owner control must be maintained.

There will be cases where a referenced object must be removed for reasons orthogonal to referential integrity. Any proposed referencing mechanisms should remove this kind of ultimate control from object owners.

The system must be robust.

The conflict between correctness and robustness has to be acknowledged (blocking forever cannot really be tolerated); complex integrity-preservation protocols may not operate in the large-scale, heterogeneous, failure-prone world of the Internet. One of the factors contributing to the success of the web is the simplicity of the design; stateless system (clients, servers and protocol).

Pragmatic Refinements

One of the main issues here is to augment the basic referencing design with features that improve the robustness of the system. Approaches include:

  • Collections - structuring technique for organising information, minimises number of refs
  • Proxies - reduce the number of inter-host references
  • Alternate reference types - e.g. notification facilities via call-back using a 2-way reference

Notes

URL to Object reference translation

Since we cannot replace all URLs with object references, we must deduce the correct way for URL->reference translation to be done. The shadows' feature of allowing a reference to contain multiple paths to the resource (is this implemented?) needs to be considered; it's my feeling we'll need to use this technique extensively.

Client-stub, server-stub matching (?)

In the current implementation, although this may have now changed in Shadows, referencing is achieved by pure reference counts, there is no identity associated with a reference. This has the potential to cause problems as multiple 'de-reference' messages via the same reference may cause an object to be incorrectly collected. I have refreshed my memory of Shapiro's work and SSPs have this identity property, in that stubs are 'matched' with their associated scions. From the W3Objects point of view this is a good property.

Minimising inter-host references

We have spoken about using proxies to minimise the number of inter-host references. Basic SSP also provides this property as the default. Every reference contained in space A for object o, residing in space B, passes through the same outgoing stub. That is, in any given space there is only one stub per object in any remote space. However, every incoming reference has a separate scion.

Comparisons with Competing Technology

Hyper-G

Hyper-G from Graz, Austria is a fully functioning rival to the web (in fact it was in existence prior to Tim's creation). Among the nice features present in Hyper-G are support for referential integrity, links from arbitrary resources, user-authenticated access, and efficient searching. Hyper-G's links, which are bi-directional, are held separate from the resources within which they are held in local databases. Referential integrity is maintained locally through the use of the database, if a resource is removed, links to that resource are also removed. In the wider scale, link update information is propagated between servers using the p-flood algorithm which claims to scale well.

IETF Uniform Resource Name Proposals

The decisions about URNs have currently stalled. A number of alternate mechanisms have been proposed and are currently being implemented for evaluation. The IETF is not proposing to specify a single scheme, rather a number of schemes which all provide (at least) the same lowest-common-denominator functionality.

URNs are seen to be the solution to migration transparency. We must compare our scheme and highlight the advantages it offers: distributed-responsibility, removes name-server bottlenecks, provides migration transparency with minimal overhead, also addressing referential integrity problem.

Likely Criticisms

The most likely criticism that will be raised will be scaleability. We will have to work hard to prove that the proposed mechanisms will scale to a system as large as the current web, which will only grow over time. How can we prevent the inter-object referencing descending into an unmanageable chaos. Will an object reference count ever descend to zero in reality? Are hierarchical or domain based solutions required?
